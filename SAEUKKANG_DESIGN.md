# 새우깡으로 한계돌파 — Game Design Document

> 이 문서는 게임의 현재 사양을 기록한다. 기능이 바뀌면 변경이력에 한 줄을 추가하지 말고 이 문서의 해당 섹션을 직접 수정한다.

## 1. 개요

- **타이틀:** 새우깡으로 한계돌파
- **장르:** 2D 탑다운 로그라이크 불릿 헤븐 (Survivors-like)
- **플랫폼:** HTML5 단일 파일 (`index.html`), PC & 모바일 대응 (`?m=1`로 모바일 강제)
- **핵심 판타지:** 근육질 비둘기가 새우깡을 마구 투척하며 50개 챕터를 돌파한다 — 코스튬으로 자신을 꾸미고 가챠로 등급을 모은다

## 2. 핵심 루프

1. 챕터 진입 → 웨이브 1 → 자동 사격으로 적 처치, XP·드롭·코인 획득
2. XP 누적 → 레벨업 시 3장 카드 중 1장 선택
3. 일정 시간 경과 → 웨이브 2 → 웨이브 3 (밀도와 종류가 점차 강해짐)
4. 웨이브 3 종료 → **챕터 보스** 출현. 보스 격파 시 챕터 클리어 + 코인·코스튬 보상
5. 챕터 클리어 → 챕터 선택 화면으로 복귀, 다음 챕터가 hero 카드로 노출
6. 50번째 챕터(`무한 너머`) 보스 격파 → **시네마틱 엔딩** (전 세계 정복)

## 3. 게임 상태 & 씬 흐름

```
title ─▶ chapterSelect ─▶ playing ─▶ levelup ─▶ playing
   │                          │       │
   ├─▶ shop (skill/costume/skin)      └── paused
   │                          ├─▶ chapterCleared ─▶ chapterSelect | victory
   │                          └─▶ gameover ─▶ title | retry
   └─▶ victory (시네마틱) ─▶ title
```

타이틀과 챕터선택·상점에서도 비둘기 애니메이션이 살아있도록 `state.time`이 진행된다.

## 4. 플레이어 캐릭터: 근육질 비둘기

### 비주얼 사양

`drawPigeon()`이 매 프레임 캔버스 네이티브 드로잉으로 그린다. 타이틀 비둘기도 동일한 함수가 캔버스(`#titlePigeon`)에 렌더링하므로 코스튬이 즉시 반영된다.

### 코스튬 레이어 순서 (중요)

`drawPigeon` 안에서 z-순서가 정확히 잡혀 있다:

1. 그림자 → 다리 → 몸통 → 가슴 깃털
2. **몸통 코스튬** (옷)
3. 양팔 → 머리 → 부리 → 눈 → 눈썹 → 머리털
4. **머리 코스튬** (모자)
5. 어택 오라
6. (drawPigeon 종료 후) **무기 코스튬** (주먹 위)

이 순서 덕분에 옷이 부리·눈을 가리지 않고, 모자는 얼굴 위에 자연스럽게 얹힌다. 캔버스 변환은 `getTransform`으로 캡처해서 코스튬 호출 시 화면 좌표계로 복귀했다가 다시 비둘기 좌표계로 복원하는 방식이다.

### 웨이브 풀

`waveEnemyPool(wave)`에서 웨이브 1=가벼움, 2=중량감, 3=공세 직전. 챕터 번호에 따라 풀에 위 적이 추가된다. ch30+에는 tyrant×2, ch50에는 tyrant×3까지 누적된다.

### 밀도 & 보상 재밸런스 (탕탕특공대 스타일)

화면 가득 잡몹이 쏟아지는 타격감을 위해 스폰 레이트와 적 스탯을 동시에 조정했다.

- **스폰 레이트** (`spawnRateForWave`): 웨이브 1/2/3 = 3.6 / 5.6 / 8.0 마리/초 + 챕터당 +2.4% (이전: 1.6 / 2.6 / 3.8 + 2.0%/ch — 약 2× 밀도)
- **적 HP** ×0.65, **공격력** ×0.75 (`spawnEnemy`의 `ENEMY_HP_MULT/DMG_MULT`)
- **XP**는 `e.xp` 자체는 옛 베이스값을 유지하고, **gem.value × 0.47** (`GEM_XP_SCALE`)로 분수 스케일을 적용. 이렇게 한 이유는 `Math.max(1, Math.ceil(...))` 플로어가 mouse(XP=1)를 절반으로 내리지 못해 단순 곱은 +13% 초과를 일으켰기 때문이다.
- 결과: 웨이브 1+2+3 누적 XP는 옛 ~1934 ↔ 새 ~1936으로 거의 동일 (오차 0.1%). 한 발 한 발이 즉시 펑펑 터지고 화면이 분주하지만 레벨업 페이싱은 옛 시스템과 같다.

### 보스 — 6티어 시스템 (5챕터 간격)

웨이브 3 종료 시 1마리 출현. **챕터 구간별로 보스 종류가 다르다**. 같은 베이스 HP 공식 `(3500 + c^1.45 × 110) × chm.hp`에 변종 배수가 곱해진다.

**HP 공식 진화:**
- v0: `220 + c^1.45 × 46` — 만빌드가 0.4~8초에 보스를 폭발시킴 (의미 없음)
- v1: `700 + c^1.45 × 110` — 만빌드 ~20초, 너무 짧음
- v2: `1100 + c^1.45 × 130` — 5카드 빌드 가정. 그러나 실제로는 Lvl 15+ 14카드까지 가버림.
- **v3 (현재): `3500 + c^1.45 × 110`** — XP 복구 후 실 측정한 평균 빌드(14카드, ~100 effective DPS) + 외침 ult 시 ~30초. 사용자 요청 "최소 30초" 충족.

**검증 (ch1):**
- HP = 3500 + 110 = 3610 × 1 (chm.hp ch1) × 1 (cat_king) = **3610 hp**
- 평균 빌드 effective DPS ≈ 100 (3-4 dmg + 1-2 proj + 1-2 size + chm 1.0× 챕1 buff)
- 외침 사용 시 보스 처치 시간 = 3610 × 0.7 / 100 = **25-30초** ✓
- 외침 미사용 시 = 3610 / 100 = **36초**
- 새 플레이어 약체 빌드 (50 DPS): 외침 포함 50초 — 패턴 학습용 적정 길이

| 변종 | 챕터 | HP × | DMG × | 크기 × | 특수 패턴 |
| --- | --- | --- | --- | --- | --- |
| **왕관 고양이왕** (cat_king) | 1–29 | 1.00 | 1.00 | 1.00 | 돌진 + 3발 털뭉치 부채꼴 |
| **폭군 군주** (tyrant) | 30–34 | 1.20 | 1.10 | 1.05 | 돌진 + 반경 260 쇼크웨이브 (0.6s 텔레그래프) |
| **심해 군주** (leviathan) | 35–39 | 1.35 | 1.10 | 1.10 | 돌진 + 5발 부채꼴 호밍 촉수 (0.5s 텔레그래프) |
| **환영의 군주** (phantom) | 40–44 | 1.55 | 1.18 | 1.00 | 3.5초마다 텔레포트 + 1.8초마다 3연 다크 웨이브 |
| **잿불 불사조** (phoenix) | 45–49 | 1.80 | 1.25 | 1.10 | 10발 화염 링 AOE + **HP 50% 도달 시 1회 부활(60% HP)** |
| **공허의 황제** (void) | 50 | 2.80 | 1.45 | 1.20 | 멀티 페이즈 — 1단계 8발 보이드 블래스트, 2단계(66% HP)에 12발 + 텔레포트, 3단계(33% HP)에 14발 + 회전 + 4 다크 위성 |

Ch48, 49는 phoenix를 그대로 사용하므로 자동으로 Ch50의 void emperor보다 약하다.

보스가 화면 밖에 있으면 화면 가장자리에 빨간 화살표 + `보스 N m` 거리 라벨이 떠서 위치를 알려준다. 보스 출현 후 처음 3초 동안 보스 머리 위에 변종 이름 라벨이 표시된다.

### 보스 처치 시간 설계 (재밸런스)

타겟: **레벨 캡 직전(현재 챕터 만렙) + 외침(30% HP 깎음) + 명중률 60%** 가정에서 보스전 ≥ 30s. 첫 시도(빌드 미완성)는 60s+, 만빌드(상점 만렙 + S 코스튬 풀세트)도 외침 포함 ~25–30s가 되도록 베이스 HP를 (220 + c^1.45×46) → (700 + c^1.45×110)으로 ~3× 부풀렸다.

규칙: 외침 사용 후 남은 보스 HP의 70% × 변종 배수가 처치 대상.

이전 설계는 만빌드가 0.4~8s 만에 보스를 폭발시켜 패턴이 의미 없었다. 새 설계는 보스가 텔레포트/페이즈/촉수 같은 패턴을 모두 한 번씩 강요한 뒤 잡힌다.

### 챕터 시간 캡 + 실제 플레이 시간 (`chapterWaveTimes`, `chapterPlayTimeEstimate`)

각 챕터의 플레이 시간 = **웨이브 합계(고정) + 보스전(빌드 의존)**. 캡은 절대 상한이며 보스가 캡을 넘기면 강제 처치.

| 챕터 범위 | 웨이브 1개 | 웨이브 합계 | 보스 여유 | **실제 (만빌드)** | **실제 (보통빌드)** | **실제 (첫시도)** | 캡 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 1~10 | 80s | **4분** | 2분 | ~4분 | ~4-5분 | ~4-5분 | 6분 |
| 11~25 | 130s | **6:30** | 2:30 | ~6:30 | ~7분 | ~7-8분 | 9분 |
| 26~40 | 170s | **8:30** | 4분 | ~8:30-9분 | ~9-10분 | ~9-12분 | 12:30 |
| 41~50 | 200s | **10분** | 5분 | ~10-11분 | ~11-13분 | ~11-14분 | 15분 |

**Ch1~Ch50 만빌드 1회 클리어 누적**: 약 6시간 30분.

**노출 위치**:
- 챕터 선택 hero 카드: ⏱ 예상 플레이 X-Y분 라벨
- 인게임 HUD: 현재 웨이브 남은 시간 (mm:ss)
- 챕터 클리어 화면: 생존 시간 N초 (실제 측정)
- 조작법 모달: 챕터 범위별 플레이 시간 + 캡 안내

### 난이도 배율 (`chapterMult`)

5단 piecewise 곡선:
- Ch 1~10: 선형 +0.26 hp / +0.08 dmg
- Ch 10~20: 지수 1.09/1.04
- Ch 20~30: 지수 1.13/1.05

#### 등급별 시각 차별화 (3축 누적)

같은 외형도 등급에 따라 시각적으로 명확히 다르다. 세 축으로 누적 차별화:

**1. 사이즈 차이 (`tierScaleFor`)**
- C: ×0.80 (가장 작음, 입문용)
- B: ×0.95
- A: ×1.12
- S: ×1.30 (가장 큼, 최고급)

머리·무기 코스튬은 전체 스케일이 적용되고, 망토 등 몸통 코스튬의 외부 돌출부(뒤판)도 스케일이 적용된다. 몸통 메인은 비둘기 실루엣 클립 때문에 사이즈는 그대로지만 디테일/후광이 변한다.

**2. 후광 (S/A 전용)**
- C/B: 후광 없음
- A: 단일 솔리드 후광 (반경 ×1.15)
- S: 외곽 펄스 후광(반경 ×1.6) + 내부 후광(반경 ×1.2) — 시간에 따라 0.7~1.0 펄스

**3. 디테일 누적**
- C: 베이스 형태만 (단색, 미니멀)
- B: + 1~2 작은 액센트 (밴드, 스터드)
- A: + 3~4 장식 (보석, 트림, 패치)
- S: + 애니메이션 효과 (회전 별, 펄스, 불꽃, 스파클)

**예시:**
- **h_crown**: C 단순 띠 → B 5점 왕관 → A + 가운데 빨간 보석 → S + 양옆 보석 + 회전 별 + 후광
- **h_beanie**: C 폼 없는 회색 도ome → B 색상+작은 폼 → A 줄무늬+큰 폼 → S 회전 스파클+큰 사이즈+후광
- **b_cape**: C 짧은 회색 뒤판 → B 색상+클래스프 → A 메달리온+상의 트림 → S 거대 황금 망토+애니메이션 흐름+림 스파클
- **b_dragon**: C 회색 비늘 → B + 금색 척추 → A + 배 엠블럼 → S + 비늘 색상 시프트 + 불꽃 위스프
- **w_phoenix**: C 단일 코어 → B + 3 꽃잎 → A + 5 꽃잎 회전 + 후광 → S + 외곽 링 + 카운터 회전 스파클 + 펄스 후광

## 14. UI 레이아웃

- `#title` — 타이틀 화면. 우상단 🛒 상점 핀, 중앙에 코스튬 입은 비둘기 캔버스, 챕터 선택 / 조작법 / 기록 초기화 버튼
- `#chapterSelect` — 포커스 카드 + 정복 스트립 + 다가올 스트립 (또는 전체 보기 그리드)
- `#shopScreen` — 코인 바 + 3 탭(스킬/코스튬/스킨)
- `#hud` — 인게임 (위에 설명)
- 모달: `levelUp`, `pauseModal`, `chapterCleared`, `gameOver`, `victory`, `gachaModal`, `equipPickerModal`, `costumePreviewModal`, `skinPreviewModal`, `howModal`

PC(≥768×700)에서는 오버레이 컨텐츠가 수직 중앙 정렬된다 (`::before`/`::after` flex 스페이서). 모바일은 top-anchored. 백 버튼은 `position:fixed`로 항상 좌상단에 고정.

### 모달 스태킹 규칙

- `.modal`은 `position:fixed; z-index:1000`으로 뷰포트 전체를 덮는다.
- 백 버튼(`.back-btn`)은 `position:fixed; z-index:5` — 일반 상황에서는 보이지만 모달이 떠있으면 가려져야 한다.
- z-index만으로는 Chrome이 일부 뷰포트에서 `position:fixed` 백 버튼을 모달 백드롭 위에 페인트하는 문제가 있어, **MutationObserver**로 어떤 `.modal`이라도 `.hidden`이 풀리면 `body.modal-open` 클래스를 토글한다. CSS는 이 클래스가 있을 때 `.back-btn { visibility:hidden }`로 강제로 가린다. 클릭 흡수와 시각 일관성 모두 확보.

## 15. 배경 (탑뷰 전용)

이 게임은 위에서 내려다보는 탑뷰 로그라이크다. **측면뷰 요소는 절대 사용 금지** (지평선, 세워진 나무, 건물 정면, 깃발 등). 모든 환경 오브젝트는 위에서 본 모습으로 그린다.

### 렌더링 파이프라인

1. **티어 HUE 그라디언트** (screen) — 분위기 컬러 베이스
2. `ctx.translate(-camX, -camY)` — 월드 좌표로
3. **`drawArenaFloor`** (world) — 타일 색 + 그리드 + 타일별 프롭
4. 젬 → 아이템 → 회오리 → 적 → 적탄 → 플레이어(코스튬 포함) → 플레이어 탄 → 파티클 → 데미지 숫자
5. `ctx.restore()` — 스크린 복귀
6. **`drawSkyline`** (screen 오버레이) — 날씨 파티클 + 비네트 + 중앙 ambient glow
7. 보스 경고 배너 + 보스 방향 화살표

### 티어별 테마
